## 1. 프로젝트 개요

### 1-1) BookFinder란?
- Vite + React(TypeScript) 기반 책 검색 웹 앱
- 카카오 Books API를 사용해 책을 검색하고 상세 정보를 표시
- 환경 변수 미설정 시 mock 데이터로 자동 폴백되어 언제든 로컬 실행 가능

### 1-2) 주요 목표
- React 상태 관리 패턴 학습: Hook ↔ Context ↔ 컴포넌트 데이터 전달
- 실제 API 통합으로 비동기 처리 경험
- TypeScript를 활용한 타입 안전성 확보

### 1-3) 빠른 시작
```bash
npm install
npm run dev        # 개발 서버: http://localhost:5173
npm run build      # 프로덕션 빌드
```

환경 변수 설정(선택):
```env
VITE_KAKAO_REST_API_KEY=여기에_키를_넣으세요
```
미설정 시 mock 데이터 사용

## 2. 핵심 파일 구조

### 2-1) 타입 레이어
- `src/types/book.ts`: Book, BookMeta, BookSearchResponse 인터페이스 정의
- API 응답과 컴포넌트 Props 간 계약을 명확히 함

### 2-2) 데이터 레이어
- `src/data/mockResponse.ts`: API 키 미설정 시 사용할 mock 책 목록과 메타데이터

### 2-4) UI 레이어 (컴포넌트)

```
│   ├── BookList (Book 배열 표시)
```

## 3. 데이터 흐름 (API → Hook → Context → 컴포넌트)

### 3-1) 흐름 개요
```
사용자 검색 입력
    ↓
SearchBar.tsx (입력값 전달)
    ↓
useSearch.ts (handleSearch 실행)
    ↓
API 호출 OR mock 데이터 반환
    ↓
SearchContext 업데이트
    ↓
SearchArea, BookDetail 리렌더링
```

### 3-2) 단계별 동작
1. **사용자가 검색창에 입력** → SearchBar에서 `onSearch()` 콜백 호출
2. **useSearchContext().handleSearch(keyword)** 실행
3. **useSearch.ts 내부에서**:
   - `setQuery(keyword)`로 상태 업데이트
   - `page` 초기화 (1로)
   - `fetchBooks()` 실행: API 키 있으면 Kakao API 호출, 없으면 mock 사용
   - `setBooks()`, `setMeta()`, `setLoading()` 갱신
4. **Context 값 변경** → 의존하는 컴포넌트 자동 리렌더링
5. **BookList가 새 books 배열 표시**, 선택된 BookDetail도 동시에 볼 수 있음

### 3-3) 선택된 책 보존 (중요 UX 포인트)
- 페이지네이션이나 새 검색 시에도 이미 선택된 책은 유지됨
- 처음 검색 결과일 때만 자동으로 첫 책을 선택
- 코드: `useSearch.ts`의 `useEffect` 의존성 배열과 선택 로직 참고

## 4. Custom Hook (`useSearch`) 정리

### 4-1) Hook이 뭘까?
- React 함수형 컴포넌트에서 상태와 부수효과를 다루기 위한 함수
- `useState`, `useEffect`, `useContext` 등을 조합해 로직을 재사용 가능한 형태로 추상화

### 4-2) 이 프로젝트의 useSearch
**위치**: `src/hooks/useSearch.ts`

**반환 값(State + 메서드)**:
```typescript
{
  query: string,           // 검색 키워드
  page: number,            // 현재 페이지 (1-indexed)
  books: Book[],           // 검색 결과 책 목록
  meta: BookMeta,          // 검색 결과 메타 (total_count, is_end 등)
  selectedBook: Book | null,  // 사용자가 선택한 책 (null이면 미선택)
  loading: boolean,        // API 호출 중 여부
  error: string | null,    // 에러 메시지
  
  // 메서드들
  handleSearch: (keyword: string) => void,  // 검색 시작
  setPage: (p: number) => void,             // 페이지 변경
  setSelectedBook: (b: Book | null) => void // 선택 책 변경
}
```

### 4-3) 키 로직: fetchBooks()
```typescript
const fetchBooks = async (searchQuery: string, searchPage: number) => {
  setLoading(true);
  setError(null);
  
  try {
    const response = await cakaoBooksAPI(searchQuery, searchPage);
    // OR if no API key: use mockResponse
    
    setBooks(response.documents);
    setMeta(response.meta);
    
    if (!selectedBook && response.documents.length > 0) {
      setSelectedBook(response.documents[0]);  // 초선택
    }
  } catch (e) {
    setError(String(e));
  } finally {
    setLoading(false);
  }
};
```

### 4-4) useEffect로 자동 패칭
- 의존성: `[query, page]`
- 즉, query 또는 page가 바뀔 때마다 자동으로 `fetchBooks()` 실행
- 사용자가 SearchBar에서 검색하거나 Pagination에서 페이지 변경 → 자동 호출

## 5. Context API와 Prop Drilling 문제

### 5-1) Prop Drilling이란?
- Props를 컴포넌트 트리의 깊은 곳까지 전달해야 할 때, 중간 컴포넌트들이 props를 받고 또 전달하기만 하는 상황
- 복잡도 증가, 유지보수 어려움

예시 (WITHOUT Context):
```jsx
<BookFinder books={books} onSelectBook={...}>
  <SearchArea books={books} onSelectBook={...}>
    <BookList books={books} onSelectBook={...}>
      <BookItem book={item} onSelectBook={...} />
    </BookList>
  </SearchArea>
</BookFinder>
```

### 5-2) Context API로 해결
```jsx
<SearchProvider>  {/* SearchContext.Provider 역할 */}
  <BookFinder>
    <SearchArea>
      <BookList>
        <BookItem />  {/* useSearchContext() 직접 사용, props 불필요 */}
      </BookList>
    </SearchArea>
  </BookFinder>
</SearchProvider>
```

### 5-3) 이 프로젝트의 구조
- `SearchContext.tsx`: `useSearch` 반환 값을 Context로 제공
- `BookFinder.tsx`: `<SearchProvider>`로 전체 UI를 감싸기
- 모든 자식 컴포넌트: `useSearchContext()`로 한 번에 필요한 모든 데이터/메서드 접근

**Prop Drilling 따라가기 불필요** → 매직!

### 5-4) Context 사용 시 주의
- Context는 값이 바뀔 때마다 의존 컴포넌트 전체를 리렌더링
- 변경 빈도가 매우 높으면 성능 저하 가능 (이 프로젝트는 문제 없음)
- 모든 전역 상태를 Context에 넣을 필요는 없음 (분리 기준: "얼마나 많은 컴포넌트가 필요한가?", "얼마나 자주 바뀌는가?")

## 6. 타입 체계와 mock 데이터

### 6-1) 왜 타입 선언?
- TypeScript 컴파일 시점에 오류 감지 (런타임 전)
- IDE 자동완성 지원
- API 응답 구조 문서화
- 리팩터링 안전성 향상

### 6-2) Book 타입 정의 (src/types/book.ts)
```typescript
export interface Book {
  title: string;
  thumbnail: string;
  status: string;
  isbn: string;
  authors: string[];
  publisher: string;
  datetime: string;
  contents: string;
  url: string;
}

export interface BookMeta {
  total_count: number;
  pageable_count: number;
  is_end: boolean;
}

export interface BookSearchResponse {
  documents: Book[];
  meta: BookMeta;
}
```

### 6-3) Mock 데이터 (src/data/mockResponse.ts)
- Kakao API 응답 구조와 동일하게 작성
- API 키 없거나 네트워크 오류 시 사용
- SearchBar 입력값으로 `title` 필터링 (간단한 로컬 검색)

### 6-4) useSearch에서의 폴백 로직
```typescript
if (apiKey) {

} else {
  // mock 사용 + 제목 필터링
```

## 7. 한 줄 요약

`useSearch` Hook이 API 호출이나 mock 데이터를 관리하고, `SearchContext`가 모든 하위 컴포넌트에 상태를 공급하므로, Prop Drilling 없이 BookFinder, SearchArea, BookDetail 등에서 같은 검색 상태에 접근할 수 있다.
